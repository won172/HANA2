# HANA Budget Skill

## 이 skill의 목적
이 skill은 `HANA2` 프로젝트를 수정하거나 확장할 때 AI가 반드시 알아야 할 공통 규칙을 정리한 문서다.

이 프로젝트는 단순 결제 앱이 아니다.  
**목적형 예금토큰 기반 예산 운영 플랫폼 MVP**이며, 핵심은 아래 흐름을 안정적으로 운영하는 것이다.

- 예산 신청
- 승인 / 반려
- 예산 발행
- 정책 기반 집행
- 정산 / 종료
- 감사 기록

즉, 이 프로젝트는 **기관용 목적형 자금 집행 플랫폼**을 캠퍼스 예산 시나리오로 실증하는 형태다.

---

## 프로젝트 정체성
이 프로젝트를 이해할 때 가장 먼저 기억해야 할 것은 아래 4가지다.

1. 이 서비스의 중심은 **정책 엔진(policy engine)** 이다.
2. 잔액 변화의 기준 데이터는 **LedgerEntry** 다.
3. 주요 사용자 흐름은 **club / admin** 이다.
4. 블록체인은 결제 엔진이 아니라 **감사 레이어** 다.

---

## 절대 깨면 안 되는 핵심 원칙

### 1. 정책 엔진은 UI보다 우선한다
화면에서 승인처럼 보여도 실제 판정은 반드시 정책 엔진을 기준으로 한다.

- APPROVED
- NOTIFIED
- PENDING
- DECLINED

이 4개 상태의 의미를 바꾸지 않는다.

### 2. LedgerEntry는 잔액 변동의 단일 기준이다
잔액 변화는 `Budget.currentBalance`만 보고 처리하지 않는다.  
반드시 `LedgerEntry`와 함께 생각한다.

다음 이벤트는 모두 원장에 남아야 한다.

- 예산 발행
- 승인된 집행
- 환불
- 만료 환수

### 3. POS는 보조 도구다
`/pos`는 데모를 위한 시뮬레이터다.  
실제 사용자 흐름의 중심은 아니다.

실제 운영 흐름은 아래 기준으로 본다.

- 동아리 / 학생회 사용자: `/club`
- 관리자: `/admin`
- POS: 보조 데모 도구

### 4. AI는 보조 판단 역할이다
AI는 정책 엔진을 대체하지 않는다.

AI의 역할:
- 카테고리 추천
- 리스크 점수 보조
- 설명/사유 초안
- 이상 패턴 요약

AI가 직접 승인 기준이 되어서는 안 된다.

### 5. 블록체인은 감사 가능성 레이어다
이 프로젝트에서 블록체인은 결제 수단이 아니다.

블록체인의 역할:
- 핵심 이벤트 해시 앵커링
- 정책 스냅샷 위변조 방지
- 승인/반려 결정 추적성 보강
- 정산 보고 신뢰성 강화

## 주요 사용자와 목적

### 관리자(Admin)
관리자는 아래 일을 한다.

- 예산 신청 검토
- 예산 발행
- 보류 거래 검토
- 정산 검토
- 가맹점 관리
- 감사 기록 확인

관리자 화면은 **운영 도구**처럼 보여야 한다.  
단순 통계판처럼 보이면 안 된다.

### 동아리 / 학생회 사용자(Club)
동아리 사용자는 아래 일을 한다.

- 예산 신청
- 배정 예산 확인
- 정책 확인
- 집행 요청
- 보류 거래 보완
- 정산 보고

동아리 화면은 **업무 시작 화면**처럼 보여야 한다.

### POS
POS는 실제 가맹점 단말이 아니라 데모용 시뮬레이터다.  
동아리 화면에서 요청하는 흐름이 더 중요하다.

---

## 주요 도메인 모델 요약

### Organization
예산의 주체다.

예:
- 학교
- 학생처
- 총학생회
- 통계학과 학생회
- 데이터분석 동아리

### User
로그인/접속 주체다.

### BudgetRequest
동아리/학생회가 먼저 제출하는 예산 신청서다.

### Budget
승인 후 발행된 예산이다.

### Policy
예산이 어떻게 쓰일 수 있는지를 정의하는 규칙이다.

예:
- 허용 카테고리
- 금지 키워드
- 자동 승인 한도
- 수동 검토 한도
- 신규 가맹점 허용 여부
- 시간대 제한

### Transaction
집행 요청 단위다.  
승인/알림/보류/거절 상태를 가진다.

### LedgerEntry
예산 잔액 변동 내역이다.  
실제 회계 추적의 기준이다.

### BudgetSettlement
정산 및 종료 보고 단위다.

### Merchant
가맹점 운영 관리 단위다.

### AnchorRecord
감사용 앵커링 이벤트 기록이다.

## 현재 프로젝트에서 우선시해야 하는 흐름
아래 순서가 자연스럽게 연결되어야 한다.

1. 동아리가 예산 신청서를 작성한다.
2. 관리자가 신청을 승인 또는 반려한다.
3. 승인되면 예산과 정책이 발행된다.
4. 동아리가 정책을 확인하고 집행 요청을 보낸다.
5. 정책 엔진이 요청을 판정한다.
6. 보류 건은 관리자 검토 또는 사용자 보완으로 이어진다.
7. 예산 종료 시 정산 보고를 제출한다.
8. 핵심 이벤트는 감사 레이어에 기록된다.

기능을 추가할 때는 아래 우선순위를 따른다.

### 1순위

- 예산 신청
- 승인 / 반려
- 발행
- 집행
- 정산

### 2순위
정책 가시성 및 UX

- 사유를 사용자에게 친절하게 노출하기
- 보완/재요청 흐름 연결하기

### 3순위
감사 가능성 강화

- AnchorRecord
- 정책 스냅샷
- 정산 보고 해시
- 관리자용 감사 화면

### 4순위
AI 운영 보조

- 이상 패턴 탐지
- 운영 요약
- 사유 생성 보조

## 변경할 때 조심할 것

### 하지 말아야 할 것
- 기존 구조를 이유 없이 크게 갈아엎기
- policy engine 로직을 페이지 컴포넌트 안으로 흩뿌리기
- currentBalance만 믿고 원장 무시하기
- POS를 메인 플로우처럼 만들기
- AI를 승인 기준으로 격상시키기
- 블록체인을 전 거래 저장소처럼 다루기

### 권장하는 방식
- 기존 페이지/컴포넌트 최대한 재사용
- DB 스키마 변경은 최소화
- seed와 demo flow를 항상 같이 맞추기
- 새 기능은 운영 흐름 안에 자연스럽게 삽입하기

## UI/UX 공통 기준
이 프로젝트의 UI는 화려함보다 **운영성**이 중요하다.

### 관리자 화면
관리자 화면은 아래가 바로 보여야 한다.

- 지금 검토할 신청
- 지금 검토할 보류 거래
- 만료 임박 예산
- 정산 대기 건
- 최근 감사 이벤트

### 동아리 화면
동아리 화면은 아래가 바로 보여야 한다.

- 현재 남은 예산
- 사용 가능 정책
- 지금 요청할 수 있는 작업
- 보류/반려 사유
- 정산 해야 하는 예산

### 사유 문구
사유는 짧고 행동이 이어져야 한다.

좋은 예:
- 자동 승인 한도를 초과해 관리자 검토가 필요합니다.
- 금지 키워드가 포함되어 요청이 반려되었습니다.
- 신규 가맹점이므로 추가 설명 제출 후 재요청하세요.

---

## 완료 기준
작업 완료는 단순히 페이지가 뜨는 것이 아니다.

아래가 함께 맞아야 완료다.

- 페이지가 렌더링된다.
- 관련 API가 연결된다.
- DB 데이터가 저장된다.
- seed 데이터로 재현된다.
- club/admin 시나리오가 이어진다.
- 정책 엔진과 원장이 깨지지 않는다.

---

## 작업 시작 시 AI가 따라야 할 절차
새 작업을 시작하면 반드시 아래 순서를 따른다.

1. 현재 작업이 운영 흐름 어디에 속하는지 파악한다.
2. 관련 도메인 모델을 먼저 확인한다.
3. policy engine / ledger / request / settlement 영향 여부를 점검한다.
4. 수정할 파일을 최소 범위로 잡는다.
5. API, UI, seed까지 함께 맞춘다.
