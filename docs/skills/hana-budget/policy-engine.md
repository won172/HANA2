# HANA Budget Skill - Policy Engine

## 문서 목적
이 문서는 `lib/policyEngine.ts`를 수정하거나 확장할 때 AI가 반드시 지켜야 할 기준을 정리한다.

정책 엔진은 이 프로젝트의 핵심 비즈니스 로직이다.  
화면보다 우선하며, 예산 집행 가능 여부의 기준이 된다.

---

## 정책 엔진의 역할
정책 엔진은 집행 요청을 아래 상태 중 하나로 판정한다.

- `APPROVED`
- `NOTIFIED`
- `PENDING`
- `DECLINED`

이 상태는 단순 UI 뱃지가 아니라 **운영 의사결정 상태**다.

---

## 상태 정의

### APPROVED
즉시 승인.

조건 예시:
- 허용 카테고리
- 금지 키워드 없음
- 잔액 충분
- 기간 내 사용
- 자동 승인 한도 이하
- 승인된 가맹점 또는 허용 가능한 신규 가맹점

### NOTIFIED
승인은 하되 관리자에게 알림.

조건 예시:
- 정책에는 맞음
- 다만 금액이 자동 승인 한도를 초과
- 수동 검토 한도 이하

### PENDING
검토 필요.

조건 예시:
- 신규 가맹점
- 미승인 가맹점
- 허용 여부가 애매한 카테고리
- 행사 기간 외 사용 의심
- 제한 시간대 사용
- manualReviewLimit 초과
- 허용/금지 키워드가 충돌

### DECLINED
즉시 거절.

조건 예시:
- 잔액 부족
- 예산 만료
- 금지 카테고리
- 금지 키워드
- 정책상 명백한 위반

---

## 절대 깨면 안 되는 규칙

### 1. 정책 엔진은 순수 판정 함수에 가깝게 유지한다
페이지 컴포넌트 안에서 승인 규칙을 새로 만들지 않는다.  
판정 기준은 되도록 `policyEngine.ts`에 모은다.

### 2. reason은 반드시 사람이 이해할 수 있어야 한다
상태만 반환하면 안 된다.  
항상 사용자가 이해할 수 있는 사유 문자열을 함께 반환한다.

### 3. UI용 로직과 비즈니스 로직을 섞지 않는다
색상, 뱃지 이름, 카드 표현은 UI 레이어에서 처리한다.  
정책 엔진은 상태와 사유를 반환하는 데 집중한다.

### 4. currentBalance와 LedgerEntry는 함께 고려해야 한다
정책 엔진은 집행 가능 여부만 판단한다.  
실제 잔액 반영은 트랜잭션 처리와 원장 기록이 함께 가야 한다.

## 권장 확장 방향

### 카테고리별 세부 한도
`categoryAutoApproveRules`를 유지하며 카테고리별 자동 승인 한도를 강화한다.

### 시간대 정책
quiet hours 외에도 주말, 행사 당일, 특정 기간 조건으로 확장할 수 있다.

### 행사 윈도우 정책
행사성 예산의 오남용을 막기 위해 `eventCategories`와 기간 필드를 함께 사용한다.

### Merchant 연동 강화
- 승인된 가맹점인지
- 가맹점 기본 카테고리가 무엇인지
- 신규 등록 상태인지
를 반영한다.

---

## 테스트 관점 핵심 시나리오
다음은 항상 살아 있어야 한다.

1. 허용 카테고리 + 소액 + 승인 가맹점 → APPROVED
2. 허용 카테고리 + 자동 승인 한도 초과 → NOTIFIED
3. 신규 가맹점 + allowNewMerchant=false → PENDING
4. 금지 키워드 포함 → DECLINED
5. 예산 만료 → DECLINED
6. 잔액 부족 → DECLINED
7. 행사 카테고리지만 행사 기간 외 → PENDING
