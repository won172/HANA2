# HANA Budget MVP 고도화 작업지시서

## 1. 문서 목적
이 문서는 현재 구현된 **HANA Budget MVP**를 다음 단계로 고도화하기 위해, AI가 바로 이해하고 작업할 수 있도록 정리한 작업지시서다.

현재 서비스는 **목적형 예금토큰 기반 예산 집행 로직**을 시연하는 데모로는 충분하지만, 아직 실제 운영 서비스 수준은 아니다.  
따라서 다음 단계에서는 단순 결제 통제를 넘어 **예산 신청 → 승인 → 발행 → 집행 → 정산 → 환수**까지 이어지는 운영 흐름을 보강해야 한다.

---

## 2. 현재 구현 상태 요약

### 이미 구현된 것
- 관리자 로그인 / 동아리 사용자 로그인 / Mock POS 로그인
- 관리자 대시보드
- 예산 발행 화면
- 보류 거래 검토 화면
- Mock POS 결제 요청 화면
- 정책 엔진 기반 승인 / 보류 / 거절
- LedgerEntry 기반 원장 기록
- AI 보조 분석 카드(리스크 점수, 추천 카테고리)

### 현재 한계
- 동아리 사용자가 예산을 보다가 바로 결제를 요청하는 흐름이 약함
- 예산 운영의 앞단(예산 신청)과 뒷단(정산/종료 보고)이 없음
- POS 화면이 메인 흐름처럼 보여 실제 서비스 느낌이 약함
- 카테고리 중심 정책만으로는 실무성이 부족함
- 블록체인은 아직 연결되지 않았고, 현재는 **오프체인(DB) 원장 시스템** 상태임
- 데모 데이터의 금액 규모가 비현실적이라 신뢰감이 떨어질 수 있음

---

## 3. 이번 고도화의 핵심 목표

이번 단계의 목표는 아래 3가지다.

1. **운영 흐름 완성**
   - 예산 신청 → 승인 → 발행 → 사용 → 정산 → 환수 구조 만들기

2. **실제 사용자 경험 강화**
   - 동아리 사용자 기준으로 더 자연스럽게 예산 집행이 가능하도록 UX 개선

3. **블록체인 역할 명확화**
   - 블록체인은 결제 처리용이 아니라, 핵심 이벤트를 위변조 방지용으로 기록하는 **감사 레이어**로 설계

---

## 4. 가장 먼저 해야 할 일 (우선순위 1)

### 4-1. 데모 데이터 현실화
현재 금액 데이터가 너무 비현실적이다.  
학생회/동아리 예산처럼 아래 수준으로 수정한다.

#### 수정 예시
- 종강총회 행사 예산: 500,000원
- 홍보물 제작 예산: 300,000원
- 동아리 운영비: 1,200,000원
- 세미나 대관 예산: 800,000원

#### 해야 할 작업
- seed 데이터 수정
- 관리자 대시보드 금액도 현실적 단위로 재생성
- 그래프/잔액 카드도 함께 맞추기

---

### 4-2. 동아리 대시보드 안에서 바로 결제 요청 가능하게 만들기
현재는 동아리 사용자가 예산을 보고 다시 Mock POS로 이동해야 한다.  
이 흐름은 실제 서비스처럼 느껴지지 않는다.

#### 목표
- 동아리 대시보드 또는 예산 상세 화면 안에 **결제 요청하기** 기능 추가
- 사용자는 예산을 확인한 뒤 같은 화면에서 바로 결제 요청 가능해야 함

#### 해야 할 작업
- `/club` 또는 `/club/budgets/[id]`에 결제 요청 버튼 추가
- 결제 요청 폼은 최소 필드로 구성
  - 예산 선택
  - 가맹점명
  - 금액
  - 카테고리
  - 품목/메모
- 요청 후 결과(승인/보류/거절)와 사유를 즉시 보여주기

#### 주의
- `/pos`는 삭제하지 말고 **데모 모드** 또는 **가맹점 시뮬레이터** 역할로 위치를 낮춘다

---

### 4-3. 정책 정보 가시화
사용자는 결제 전에 어떤 예산이 어디까지 허용되는지 알아야 한다.

#### 예산 상세에 반드시 노출할 것
- 사용 가능 기간
- 허용 카테고리
- 금지 카테고리
- 금지 키워드
- 자동 승인 한도
- 수동 검토 기준
- 신규 가맹점 허용 여부
- 만료 임박 여부

#### 목적
- “왜 안 되는지”를 사후가 아니라 사전에 이해하게 만들기

---

## 5. 두 번째로 해야 할 일 (우선순위 2)

### 5-1. 예산 신청 기능 추가
지금은 관리자가 직접 예산을 발행한다.  
실제 서비스라면 동아리 쪽에서 먼저 예산을 신청해야 한다.

#### 새로 추가할 개념
`BudgetRequest`

#### BudgetRequest 필드 예시
- id
- organizationId
- title
- purpose
- requestedAmount
- requestedCategories
- requestedPeriodStart
- requestedPeriodEnd
- status (PENDING, APPROVED, REJECTED)
- reviewerComment
- createdAt

#### 사용자 흐름
1. 동아리가 예산 신청서 작성
2. 관리자가 검토
3. 승인 시 Budget + Policy 자동 생성
4. 반려 시 사유 저장

#### 화면
- `/club/requests/new`
- `/club/requests`
- `/admin/requests`

---

### 5-2. 보류 거래 후속 처리 강화
지금은 보류 거래를 승인/반려만 할 수 있다.  
실제 서비스에서는 보류 이후 사용자 행동도 필요하다.

#### 추가할 기능
- 추가 설명 제출
- 수정 후 재요청
- 반려 사유 확인
- 증빙 첨부(파일 업로드는 후순위, 우선 텍스트 메모부터)

#### 예시
- “신규 가맹점이라 검토 필요”
- “품목 설명이 불명확하여 추가 설명 요청”
- “예산 목적과 직접 연관성 부족”

---

### 5-3. 정산/종료 보고 기능 추가
실제 운영에서는 예산이 끝난 뒤 마무리 보고가 필요하다.

#### 추가할 화면
- `/club/budgets/[id]/settlement`
- `/admin/settlements`

#### 보여줘야 할 정보
- 총 발행액
- 총 사용액
- 총 반려액
- 보류 거래 수
- 환수 예정 금액
- 카테고리별 사용 비율
- 종료 보고 메모

#### 목적
이 기능이 들어가야 “통제 도구”가 아니라 “운영 시스템”처럼 보인다.

---

## 6. 세 번째로 해야 할 일 (우선순위 3)

### 6-1. 정책 엔진 고도화
현재는 카테고리 기반 정책이 중심이다.  
다음 단계에서는 아래 조건을 더 넣는다.

#### 추가할 정책
- 시간대 제한
  - 예: 23시 이후 결제는 무조건 보류
- 카테고리별 자동 승인 한도
  - 예: 식음료 30,000원 이하 자동 승인
  - 인쇄 100,000원 이하 자동 승인
- 신규 가맹점 자동 보류
- 금지 키워드 + 허용 키워드 조합
- 행사 일정 기반 허용 기간
  - 예: 행사 전날~당일만 식음료 결제 허용

#### 작업 방식
- `policyEngine.ts`를 템플릿/규칙 모듈화
- 룰을 if문 나열보다 설정 기반으로 점차 바꾸기

---

### 6-2. 정책 템플릿 추가
관리자가 매번 정책을 처음부터 만들면 비효율적이다.

#### 템플릿 예시
- 행사 예산 템플릿
- 홍보 예산 템플릿
- 교통 지원 템플릿
- 장비 구매 템플릿

#### 목표
예산 발행 시 템플릿을 선택하면 기본 정책이 자동으로 채워지도록 하기

---

### 6-3. 가맹점 관리 기능 추가
지금은 가맹점명이 문자열 입력 수준이다.  
운영성을 높이려면 `Merchant` 개념이 있어야 한다.

#### Merchant 필드 예시
- id
- name
- category
- isApproved
- createdAt

#### 기능
- 승인된 가맹점 목록 조회
- 신규 가맹점 자동 등록 후 보류
- 승인된 가맹점이면 자동 승인 가점

---

## 7. 블록체인 고도화 방향

## 7-1. 현재 상태
현재 서비스는 **오프체인(DB) 원장 시스템**이다.  
즉, LedgerEntry는 존재하지만 블록체인에는 아직 연결되지 않았다.

이 상태는 잘못된 것이 아니라, 현재 MVP의 범위를 고려하면 자연스럽다.

---

## 7-2. 블록체인의 역할
이 서비스에서 블록체인은 **결제 처리 레이어**가 아니다.  
블록체인의 역할은 **핵심 이벤트를 위변조 방지용으로 기록하는 감사 레이어**다.

### 블록체인에 올릴 대상
- 예산 발행 이벤트
- 정책 스냅샷 해시
- 승인/반려 결정 해시
- 환수/종료 보고 해시

### 블록체인에 굳이 안 올릴 대상
- 로그인 세션
- UI 상태
- 일반 조회 데이터
- 모든 거래 상세 텍스트 전체
- AI 분석 문장 원문

---

## 7-3. 구현 방향
### 1단계
DB 기반 원장은 그대로 유지

### 2단계
`AnchorRecord` 테이블 추가

#### AnchorRecord 필드 예시
- id
- eventType
- entityType
- entityId
- payloadHash
- chainStatus (PENDING, ANCHORED, FAILED)
- txHash
- anchoredAt

### 3단계
Hardhat local chain 또는 테스트넷에 `payloadHash`만 기록

### 4단계
관리자 화면에서 아래 보여주기
- 앵커링 여부
- tx hash
- 정책 스냅샷 해시
- 마지막 앵커링 시각

---

## 7-4. 블록체인 설명 문구
발표나 문서에서는 아래처럼 설명한다.

> 현재 MVP는 오프체인 정책 엔진과 원장으로 운영 흐름을 검증한 상태이며,  
> 향후 블록체인은 예산 발행·승인·환수 같은 핵심 이벤트를 앵커링하는 감사 레이어로 확장할 수 있다.

또는

> 블록체인은 결제를 처리하는 레이어가 아니라,  
> 예산 발행과 집행 의사결정의 위변조를 막는 감사 가능성 레이어다.

---

## 8. AI 고도화 방향

### 현재 AI 역할
- 리스크 점수
- 추천 카테고리
- 간단한 설명 보조

### 다음 단계에서 AI가 해야 할 일
1. 결제 메모 정규화
   - “현수막”, “배너”, “행사용 인쇄물” → `PRINT`
2. 반려/보류 사유 초안 자동 생성
3. 이상 패턴 탐지
   - 야간 반복 결제
   - 동일 금액 반복 요청
   - 행사 종료 직전 몰아쓰기
   - 특정 가맹점 편중
4. 관리자용 거래 요약 생성

### 주의
- AI는 정책 엔진을 대체하면 안 된다
- AI는 판단 보조 및 설명 보조에 집중한다

## 11. 최종 정리
현재 서비스는 **정책 엔진 데모**로는 충분히 좋은 상태다.  
하지만 실서비스가 되려면 **예산 운영 전체 흐름**과 **감사 레이어(블록체인)** 가 추가로 필요하다.

이번 고도화의 핵심은
- AI를 더 화려하게 붙이는 것보다
- 운영 흐름을 완성하고
- 블록체인의 역할을 감사 레이어로 명확히 하는 것
이다.
